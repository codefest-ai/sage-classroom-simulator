IST 505 Group Share Packet

Working Artifact Framing For Phase 3

Audience: IST 505 project group

Purpose: Provide one clean, shareable summary of the current project framing, the strongest contribution claims supported by the current build, the claims we should avoid overstating, and the practical writing split for Phase 3.

1. Project Framing

We are building a working instructional decision-support prototype for synchronous online teaching. In the current implementation, that artifact is demonstrated and formatively evaluated through SAGE, a classroom simulation and demo environment that lets us test the workflow under controlled scenarios before moving to human-subject or real-class deployment.

2. Strongest Novelty Claims

  1. Institutional context sensitivity The same dashboard and lesson scenario can produce different recommendation patterns across different university contexts, rather than assuming all online classrooms behave the same way.

  2. Composition-behavior separation The design explicitly separates demographic composition from behavioral modeling: demographics help describe who is in the room, while engagement behavior is driven by archetypes and institutional context modifiers rather than racialized assumptions.

  3. Instructor agency as a construct The Instructor Response Taxonomy treats the instructor as an active decision-maker who can ignore, acknowledge, accept, modify, or reject AI support, rather than as a passive endpoint for recommendations.

3. Claims We Should Avoid Overstating

  1. Do not claim full validation The current build supports formative evaluation and proof of concept. It does not yet support a strong claim that the system is fully validated in real classrooms.

  2. Do not oversell live Zoom integration There is prototype infrastructure for live integration, but it should not be framed as a finished production deployment or a completed marketplace-ready app.

  3. Do not frame novelty as simply “AI recommendations” Some existing systems already provide AI-generated or AI-assisted recommendations. Our stronger differentiation is context sensitivity, bias-aware modeling, and instructor agency.

4. What Is Already Built And Supported

  • Working prototype environment The project already includes a browser-based dashboard, scenario-driven classroom simulation, university presets, recommendation logic, and instructor-response logging.

  • Institutional-context evidence The current build already supports the claim that the same scenario can produce different outputs across different institutional contexts rather than assuming a one-size-fits-all classroom model.

  • Bias-aware design rationale The project has already resolved an important design question by separating demographic composition from behavioral modeling. That decision is part of the artifact story, not just a side note.

  • Evaluation posture The strongest current claim is formative evaluation and proof of concept. The current artifact supports scenario testing, comparison, and prototype demonstration; it does not yet support a strong real-world validation claim.

  • LLM-enhanced simulation The project also includes an AI mode in which students and professor behavior can be driven through an LLM-backed server path. This is best presented as an important methodological and demonstration capability, not as the sole source of novelty.

5. What The Writing Team Can Build From This

  • A cleaner novelty statement centered on context sensitivity, bias-aware modeling, and instructor agency
  • A design-principles section grounded in actual build decisions rather than abstract aspirations
  • A safer evaluation section that distinguishes current evidence from future validation
  • A Phase 3 draft that already has enough structure to assign and revise collaboratively
  • A stable artifact boundary even if the final product name remains provisional for now

6. Proposed Division Of Labor For Phase 3

  • Evren Own the artifact description, screenshots, dashboard walkthrough, simulation architecture, proof-of-concept section, and short artifact spec.

  • Writing team Own the problem statement, literature gap synthesis, competing-systems positioning, design principles, and kernel-theory-to-feature mapping.

  • Whole group Agree on one honest novelty statement, one evaluation story, and one set of claims the whole team will consistently use.
